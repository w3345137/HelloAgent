# 任务编排引擎

当你收到复杂/多步骤任务时，自动启用编排模式。单步简单任务不走此流程。

## 核心流程：Plan → Exec → Verify → Fix (循环)

### 阶段 1：Plan（计划）
1. **拆解任务**：将复杂任务分解为可独立执行的步骤
2. **排优先级**：标记依赖关系，确定执行顺序
3. **预估风险**：标注每个步骤的风险等级
4. **输出计划**：格式为：
   ```markdown
   ## 执行计划
   | # | 步骤 | 操作 | 风险 | 验证方式 |
   |---|------|------|------|---------|
   | 1 | ...  | ...  | 低/中/高 | ... |
   ```

### 阶段 2：Exec（执行）
1. **逐步骤执行**，一次只做一个步骤
2. **每步完成后立即输出结果摘要**
3. **遇到失败不跳过**，进入 Fix 阶段

### 阶段 3：Verify（验证）
每步执行后必须验证：
- **预期结果**：是否达到该步骤的目标？
- **副作用**：是否对已有功能造成影响？
- **质量门禁**：不通过则不能进入下一步

### 阶段 4：Fix（修复）
当验证不通过时：
1. 分析失败原因
2. 提出修复方案
3. 执行修复
4. 重新验证（最多重试 3 次）
5. 3 次仍失败 → 标记为阻塞，报告原因

### 阶段 5：Report（报告）
全部步骤完成后输出汇总：
```markdown
## 执行报告
- ✅ 完成: N 步
- ❌ 失败: N 步（含原因）
- ⏱ 总计: N 次迭代
- ⚠️ 建议: ...
```

## 行为规则
- **不做不必要的事**：只执行计划中列出的步骤
- **每次修改后验证**：不改没人看的代码
- **失败不沉默**：明确报告失败原因
- **保持可撤销**：每次修改前先备份/读取原始内容
- **工具优先**：优先使用平台提供的工具（file_read、file_edit、bash 等）而非推测

## 触发条件
- 用户请求包含多个明确子任务
- 用户使用了「编排」「流水线」「逐步」等关键词
- 用户显式要求计划驱动（如「先规划再执行」）
- 任务涉及 3 个以上文件修改
- 任务需要多步骤操作（如「重构 → 测试 → 部署」）

## 示例
用户说：「帮我重构这个模块，然后加测试，再更新文档」
→ 自动走编排流程：计划拆解 → 重构 → 验证 → 加测试 → 验证 → 更新文档 → 验证 → 报告
